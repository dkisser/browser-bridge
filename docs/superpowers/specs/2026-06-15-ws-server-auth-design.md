# WS-Server API Key Authentication Design

**Date**: 2026-06-15
**Status**: Approved

## Problem

The WebSocket server currently accepts all connections without authentication. Any process that can reach the server can register browsers, send commands, and receive responses. The only existing auth is a token sent in the `register` message body — but the connection itself is never authenticated, so unauthenticated clients can still send commands and sniff responses.

## Decision

Add API key authentication at the **Local-Proxy → WS-Server** boundary only. This is the only boundary that needs protection.

### Why only this boundary

- **Extension → Local-Proxy**: Always co-deployed on the same machine. No auth needed.
- **CLI → WS-Server**: Currently local-only. No auth needed (can be added later if CLI goes remote).

## Design

### Mechanism

1. An API key is manually configured by the deployer and provided to local-proxy (via `BRIDGE_API_TOKEN` env var or `~/.browser-bridge/config.json`).
2. Local-proxy sends the API key in the WebSocket handshake `Authorization` header: `Bearer <key>`.
3. WS-Server validates the header in the `open` callback. If invalid or missing, the server closes the connection immediately with code `4001` and reason `"unauthorized"`.
4. Only after successful header validation does the connection enter the message loop (register, command, response, etc.).
5. The existing `register` message token validation is **removed** — authentication moves entirely to the handshake.

### TLS enforcement

- Connections to non-localhost URLs must use `wss://` (TLS).
- Connections to `localhost` / `127.0.0.1` / `[::1]` allow `ws://`.
- If a non-localhost connection arrives over `ws://`, the server closes it with code `4002` and reason `"tls_required"`.

### API key source

- API key is generated by the deployer when setting up the WS-Server (e.g., `openssl rand -hex 32`).
- The same key is configured on the server side (via env var `BRIDGE_API_KEYS`, comma-separated for multiple keys) and on the local-proxy side (via `BRIDGE_API_TOKEN`).
- Default/fallback key `'dev-token'` is removed — no auth bypass.

## Code changes

### `apps/websocket/src/server/index.ts`

- In the `open` handler: extract `Authorization` header, validate via `AuthProvider`.
- On failure: `ws.close(4001, "unauthorized")`.
- Add TLS check: if the request URL scheme is `ws://` and host is not localhost, close with `4002`.
- Remove the token validation from the `register` event handler. The `register` event still exists to associate a `browserId` with the connection — it just no longer validates a token (auth already happened at the handshake).

### `apps/local-proxy/src/cloud-client.ts`

- Pass `headers: { Authorization: "Bearer <apiToken>" }` when creating the WebSocket client.

### `packages/shared/src/auth.ts`

- `ApiKeyAuthProvider`: accept a `Record<string, string>` or `string[]` of valid keys.
- Add a `validateHeader(authorizationHeader: string): AuthResult` method that parses `Bearer <key>` format.

### `apps/local-proxy/src/index.ts`

- Remove the `'dev-token'` fallback. If `BRIDGE_API_TOKEN` is not set and the server URL is not localhost, local-proxy exits with an error. For local development (server URL is localhost), a missing token is allowed with a logged warning.

## Out of scope

- CLI authentication (local-only for now).
- Extension ↔ Local-Proxy authentication.
- User account system or OAuth.
- Token refresh / rotation.
- Rate limiting.
