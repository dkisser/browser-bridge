# Browser-Bridge Interim Distribution Design

Date: 2026-06-15
Status: Draft

## Goal

Provide a "good enough for now" distribution path for Browser-Bridge so an early-adopter developer can go from a single shell command to a working CLI + WebSocket Server + Local Proxy + Chrome Extension in under five minutes. The design must not paint us into a corner: the future two-machine (cloud + local) deployment must be reachable by configuration changes, not refactors.

Out of scope: production managed-service deployment, Chrome Web Store publishing, signed binaries, package-manager packages (Homebrew tap, npm publish). Each is a follow-up; the design leaves room for them but does not build them today.

## Scope Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Deployment shape | Single-machine (all four components co-located) | Simplest UX for early adopters; matches "暂行" intent. |
| Future deployment | Two-machine (CLI + WS-Server remote; Local Proxy + Extension local) | Designed-in but not implemented; reached via env vars and auth flag flips. |
| Target platforms | macOS + Linux | Bun's primary audience; skips PowerShell until demand exists. |
| Extension delivery | Downloaded from GitHub Releases as a prebuilt zip | No bun/node/git required for the extension half of install; clean zip ready for Chrome's "load unpacked". |
| Runtime orchestration | Thin `bridge` shell script generated by `install.sh` | One entrypoint; no daemon plumbing; trivial to add `bridge update`/`doctor`/`logs`. |
| Bun runtime | Required, not bundled | User installs Bun separately; documented prerequisite. Adding `bun compile` standalone binaries is a future optimization. |
| Source-of-truth for code | Shallow git clone to `~/.browser-bridge/repo/` | Lets `bridge up` invoke `bun run start` against the source directly without a build pipeline. |

## Data Directory Layout

All state lives under `~/.browser-bridge/` (no sudo required).

```
~/.browser-bridge/
├── bin/bridge              # generated orchestration script
├── repo/                   # shallow clone of browser-bridge (local-proxy + ws-server source)
├── extension/              # unpacked extension zip; user loads this folder in Chrome
├── run/                    # PID files
│   ├── local-proxy.pid
│   └── ws-server.pid
├── logs/
│   ├── local-proxy.log
│   └── ws-server.log
└── version                 # installed version string (e.g. v1.2.3)
```

`bridge` symlinks into `~/.local/bin/bridge`. PATH setup is the user's responsibility; `install.sh` prints the export line if `~/.local/bin` is not on PATH.

## Install Flow

Single command (where `<org>` is the GitHub organization hosting the repo; replaced with the real value at implementation time):

```
curl -fsSL https://raw.githubusercontent.com/<org>/browser-bridge/main/install/install.sh | bash
```

`install.sh` (POSIX bash, `set -euo pipefail`). The `bridge` script is embedded inside `install.sh` as a heredoc so both files are templated together from a single source (kept in `install/bridge.sh.tmpl` during development):

1. **Prerequisite checks** — bash ≥ 4, `curl`, `unzip`, writable `$HOME/.local`. If `bun` is missing, print installation instructions and exit `BB-E001`. Bun is required because `bridge up` shells out to it.
2. **Resolve version** — default to latest GitHub Release (parsed from `/repos/<org>/browser-bridge/releases/latest`). Accept `BB_VERSION` env var to pin a specific version. Validate the version string matches `^v?[0-9]+\.[0-9]+\.[0-9]+$`.
3. **Download extension** — `curl -fsSL` the `browser-bridge-extension-${VERSION}.zip` asset and its `.sha256` sidecar from the GitHub Release for the resolved tag. Compute SHA-256 locally and refuse to continue on mismatch (exit `BB-E020`). On any HTTP error, exit `BB-E021`.
4. **Extract extension** — unzip into `~/.browser-bridge/extension/`. If already present, rename old to `extension.bak.$(date +%s)` before overwriting.
5. **Clone or update source** — shallow clone (`git clone --depth 1 --branch <tag>`) to `~/.browser-bridge/repo/`. On reinstall, `git fetch --depth 1 origin <tag> && git reset --hard <tag>` for idempotency.
6. **Install dependencies** — `cd ~/.browser-bridge/repo && bun install --frozen-lockfile`.
7. **Write artifacts** — emit `~/.browser-bridge/bin/bridge` from the embedded heredoc, symlink it into `~/.local/bin/bridge` (creating the directory if missing), and write the resolved version to `~/.browser-bridge/version`.
8. **Print next steps** — list PATH export if needed, the Chrome "load unpacked" instructions pointing at `~/.browser-bridge/extension/`, and a one-liner `bridge up`.

`install.sh` is idempotent. Re-running it upgrades in place. On any failure, `trap` removes half-extracted artifacts.

## `bridge` Command

`bridge` is a ~150-line POSIX shell script. It does not live in `apps/`; it is templated into `~/.browser-bridge/bin/bridge` by `install.sh` so it can be regenerated without touching the bun workspace.

| Subcommand | Behavior |
|---|---|
| `bridge up` | Start ws-server, wait for its port, then start local-proxy. PID files in `run/`. Logs to `logs/`. |
| `bridge down` | `kill -TERM` from PID files; `kill -KILL` after 3s if still alive; clean PID files. Missing PID files are not errors. |
| `bridge restart` | `bridge down && bridge up`. |
| `bridge status` | Print PID/port/log-tail for each service; non-zero exit if any service is down. |
| `bridge logs [name]` | `tail -f` the named log (`ws-server` or `local-proxy`); default tails both. |
| `bridge update [version]` | Re-run install.sh logic with the target version, then `bridge restart`. Default version is latest release. |
| `bridge doctor` | Verify bun on PATH, `~/.browser-bridge/repo/` present, both PID files map to live processes, ports bound, `extension/manifest.json` valid JSON, ws-server reachable. |
| `bridge uninstall` | With `--yes`: `bridge down`, `rm -rf ~/.browser-bridge`. Without: print what would be deleted and prompt. |
| `bridge version` | Print installed version + latest release version (color-coded when TTY). |

### Startup sequence (`bridge up`)

1. Verify `~/.browser-bridge/repo/` exists; if not, print "Run the install script first" and exit `BB-E002`.
2. `mkdir -p run logs`.
3. Start ws-server: `cd ~/.browser-bridge/repo/apps/websocket && nohup bun run start > "$LOG_DIR/ws-server.log" 2>&1 & echo $! > "$RUN_DIR/ws-server.pid"`.
4. Poll the ws-server port (default 8787, configurable via `BRIDGE_WS_PORT`) for up to 5 seconds. On timeout, kill the spawned process and exit `BB-E011`.
5. Start local-proxy with the same pattern.
6. Poll its port (default 3001).
7. Print summary with PIDs and log paths.

### Shutdown sequence (`bridge down`)

1. For each service in `{ws-server, local-proxy}`:
2. Read PID from file. If empty, skip with a notice.
3. `kill -TERM $PID`. Wait up to 3s checking `kill -0`.
4. If still alive, `kill -KILL $PID`. Always remove the PID file.
5. Exit 0 even if some services were already down.

### Environment variables (read by `bridge up` and propagated to children)

| Var | Default | Purpose |
|---|---|---|
| `BRIDGE_WS_URL` | `ws://127.0.0.1:8787` | ws-server URL the local-proxy connects to. Single-machine default; future remote mode overrides here. |
| `BRIDGE_WS_PORT` | `8787` | Port ws-server binds. |
| `BRIDGE_LOCAL_PROXY_PORT` | `3001` | Port local-proxy's local WS server binds. |
| `BRIDGE_API_KEYS` | _(unset)_ | When set, ws-server enables handshake auth. Single-machine mode leaves it unset. |
| `BRIDGE_API_TOKEN` | _(unset)_ | When set, local-proxy sends it as `Authorization: Bearer <token>` on handshake. |

These vars match the existing ws-server auth design (see `2026-06-15-ws-server-auth-design.md`); we are reusing them, not introducing new ones.

## Extension Packaging & Release Automation

### Release workflow

`.github/workflows/release-extension.yml`:

- Trigger: push of a tag matching `v[0-9]+\.[0-9]+\.[0-9]+`.
- Steps:
  1. Check out the repo at the tag.
  2. Verify `package.json#version` matches the tag suffix; abort otherwise (exit `BB-E030`).
  3. Verify `CHANGELOG.md` has an entry for this version; abort otherwise.
  4. `bun install --frozen-lockfile && bun run build:extension`.
  5. Run `.github/scripts/build-extension-zip.sh` (see below).
  6. `softprops/action-gh-release@v2` uploads the zip and its `.sha256` sidecar to the GitHub Release for the tag.

### `build-extension-zip.sh`

POSIX shell script. Also runnable locally for debugging. Steps:

1. Run `bun run build:extension` if `apps/extension/dist/` is not present.
2. Create a temporary staging dir `browser-bridge-extension/`.
3. Copy contents of `apps/extension/dist/` into staging (flatten; no extra dir layer inside the zip).
4. `zip -r browser-bridge-extension-v${VERSION}.zip browser-bridge-extension/`.
5. `sha256sum` the zip; write `browser-bridge-extension-v${VERSION}.zip.sha256` containing `<hash>  <filename>`.
6. Print the absolute paths of both artifacts.

### Release artifacts

```
browser-bridge-extension-v1.2.3.zip
browser-bridge-extension-v1.2.3.zip.sha256
```

### Release process (developer side)

```
# 1. Update package.json version
# 2. Write CHANGELOG.md entry
git commit -am "release: v1.2.3"
git tag v1.2.3
git push origin main --tags
# 3. Actions builds and uploads
```

## Future-Proofing for Distributed Mode

The design intentionally leaves seams for two-machine deployment:

1. **`BRIDGE_WS_URL` is an env var** — flipping it to `wss://bridge.example.com` is the only change needed to point local-proxy at a remote ws-server.
2. **Auth is already implemented** — `BRIDGE_API_KEYS` (server) and `BRIDGE_API_TOKEN` (local-proxy) land in the current release. Single-machine mode runs with both unset (auth disabled); setting them enables handshake auth with no code changes.
3. **TLS enforcement already exists** — `@my/shared`'s `isLocalhost` check makes ws-server require TLS only when not bound to localhost.
4. **`bridge` orchestrator is portable** — when `~/.browser-bridge/` is migrated to a remote host, only the `extension/` subtree stays local; `bridge` keeps working as the remote-side manager.
5. **`install.sh` accepts `--source <git-url>`** — default is `https://github.com/<org>/browser-bridge.git`; forks and internal mirrors can override.
6. **CLI hooks** — `@browser-bridge/cli` reserves a `bridge-host` subcommand slot (placeholder + TODO comment only, not implemented) for future "point CLI at a remote server" config.

## Error Codes

All user-facing errors carry a stable code prefix `BB-E`:

| Code | Meaning |
|---|---|
| `BB-E001` | Prerequisite missing (e.g. `bun` not found) |
| `BB-E002` | `~/.browser-bridge/repo/` not present (install not run) |
| `BB-E010` | Port already in use |
| `BB-E011` | Service failed to start within timeout |
| `BB-E020` | Extension zip SHA-256 mismatch |
| `BB-E021` | Download failed (network or HTTP error) |
| `BB-E030` | Release version mismatch (tag vs `package.json`) |

Codes are documented in `install/README.md`.

## Testing Strategy

| Suite | Tool | Covers |
|---|---|---|
| `tests/install/` | `bats` | `install.sh` happy path, idempotent re-install, network failure (mock HTTP server returning 404), checksum mismatch, missing bun |
| `tests/bridge.bats` | `bats` | `up`/`down`/`status`/`restart` lifecycle, PID cleanup, missing service graceful behavior, env var propagation, port-in-use detection |
| `tests/release-workflow.test.ts` | `bun test` | `build-extension-zip.sh` produces correctly-named zip with expected internal paths; `.sha256` matches |
| Smoke (manual, documented in `install/README.md`) | — | Full curl-pipe install on a fresh VM |

Existing tests in `apps/extension`, `apps/local-proxy`, `apps/websocket`, `apps/cli` are unaffected; they continue to validate the underlying components.

## Security

- **HTTPS only** for release downloads. SHA-256 sidecar verified before extraction.
- **No sudo** anywhere in `install.sh` or `bridge`. Everything writes under `$HOME`.
- **Localhost binding** — ws-server defaults to `127.0.0.1`; `bridge doctor` warns if it is found bound to `0.0.0.0`.
- **Strict shell hygiene** — `set -euo pipefail`, all variables quoted, no `eval`, no unquoted command substitution that touches user input.
- **No secrets in this design** — single-machine mode needs no API keys. When distributed mode lands, secrets are passed via env vars read by `bridge up`, never via CLI flags.
